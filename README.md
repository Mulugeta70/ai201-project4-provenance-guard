# Provenance Guard

A Flask API that detects whether a piece of text was likely written by a human
or generated by an AI. It runs two statistical signals on submitted text,
combines them into a calibrated confidence score, returns a transparency label,
supports creator appeals, and writes a structured audit log of every decision.

---

## Architecture

A submission travels through five components before the response goes back to
the caller:

```
POST /submit
  │
  ├─► Signal 1: Vocabulary & Formality Analysis
  │     Root TTR + avg word length + long-word density → vocab_score [0,1]
  │
  ├─► Signal 2: Sentence Burstiness
  │     CoV of sentence lengths → burst_score [0,1]
  │
  ├─► Confidence Scorer
  │     0.5 * vocab_score + 0.5 * burst_score → confidence [0,1]
  │
  ├─► Label Generator
  │     confidence → transparency label string
  │
  └─► Storage + Audit Log
        UUID content_id, both signal scores, label, status

POST /appeal
  │
  ├─► Storage Lookup  (404 if not found)
  ├─► Status update: "classified" → "under_review"
  ├─► Global log entry mutated in-place (appeal_reasoning populated)
  └─► Per-submission event appended ("appeal_submitted")
```

Full architecture spec, signal formulas, threshold reasoning, and edge case
analysis: see `planning.md`.

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/submit` | Analyze text; returns classification and content_id |

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{
    "content_id": "PASTE-CONTENT-ID-HERE",
    "creator_reasoning": "I wrote this myself from personal experience. I am a non-native English speaker and my writing style may appear more formal than typical.",

  }' | python -m json.tool
```


### GET /log

```bash
curl -s http://localhost:5000/log | python -m json.tool
```

Returns up to 50 most recent structured entries (pass `?limit=N` to adjust).
Each entry includes: `content_id`, `creator_id`, `timestamp`, `attribution`,
`confidence`, `vocab_score`, `burst_score`, `label`, `status`, and
`appeal_reasoning` (null until an appeal is filed, then populated in-place).

## Detection Signals and Confidence Scoring

### Why these two signals

I chose signals that measure fundamentally different properties of text so that
a failure in one doesn't automatically mean a failure in both. Signal 1
measures what vocabulary the author chose; Signal 2 measures how they
structured sentences rhythmically. An LLM that happens to use varied vocabulary
can still be caught by sentence uniformity, and vice versa.

Both are heuristic and entirely local — no API call, no model, no network
dependency. This was a deliberate choice: a provenance system that requires
sending text to a third-party classifier adds a new point of failure and a new
privacy surface. The tradeoff is lower accuracy on adversarially crafted text,
which is documented in the limitations section below.

### Signal 1 — Vocabulary & Formality Analysis (`vocab_score`)

Three sub-metrics combined with equal weight:

**Root Type-Token Ratio (RTTR)**: measures lexical diversity normalized for
text length. Low diversity → AI-like. RTTR alone is poorly discriminative for
short texts (40–60 words) because even AI-generated text uses varied vocabulary
in a single paragraph — there simply are not enough repetitions to suppress the
ratio. Including it anyway provides useful signal on longer submissions.

**Average word length**: AI-generated text gravitates toward polysyllabic
formal vocabulary ("transformative," "implications," "deployment," "leverage")
even when asked to write about everyday topics. Casual human writing uses
shorter words ("fine," "ramen," "weird," "tired"). Average word length is
stable and discriminative even at 40 words.

**Long-word density** (fraction of words ≥ 7 characters): complements average
word length. Academic and AI prose packs in polysyllabic vocabulary; colloquial
human writing avoids it. Mapping: density ≥ 40% → score 1.0.

Formula:
```
rttr_score    = 1 / (1 + rttr / 4)
len_score     = clamp((avg_word_len - 3.0) / 5.0, 0, 1)
density_score = min(long_word_density * 2.5, 1.0)
vocab_score   = (rttr_score + len_score + density_score) / 3
```

### Signal 2 — Sentence Burstiness (`burst_score`)

Measures the coefficient of variation (std / mean) of sentence lengths in
words. A high CoV means the text mixes very short and very long sentences
(human-like). A low CoV means sentences cluster around a uniform length
(AI-like). Current LLMs, without explicit instructions, tend to produce
sentences in the 12–20-word range, yielding low variance.

Formula:
```
CoV         = stdev(sentence_lengths) / mean(sentence_lengths)
burst_score = 1 / (1 + CoV * 2)     # high CoV → low AI score
```

Returns 0.5 (neutral) if the text has fewer than 3 sentences.

### Confidence scoring

```
confidence = 0.5 * vocab_score + 0.5 * burst_score
```

Equal weights reflect that neither signal is known to be more reliable than the
other. If this were deployed with real calibration data, weights would be fit
empirically; as-is, equal weight is the honest choice.

### Two real example submissions

**High-confidence case** (confidence = 0.74):

Input:
> "The system processes data efficiently. The system analyzes all input data.
> The system generates output data. The system stores all processed data.
> The system retrieves stored data quickly. The system updates existing data
> records. The system deletes outdated data automatically."

```json
{
  "vocab_score": 0.6174,
  "burst_score": 0.8613,
  "confidence": 0.7394,
  "attribution": "likely_ai",
  "label": "Very Likely AI-Generated"
}
```

Both signals fire: repetitive vocabulary (the word "system" alone accounts for
7 of the 47 tokens) and perfectly uniform sentence lengths drive both scores
above 0.6. Confidence reaches 0.74 — the highest tier.

**Lower-confidence case** (confidence = 0.39):

Input:
> "ok so i finally tried that new ramen place downtown and honestly?
> underwhelming. the broth was fine but they put WAY too much sodium in it and
> i was thirsty for like three hours after. my friend got the spicy version and
> said it was better. probably would not go back unless someone dragged me there"

```json
{
  "vocab_score": 0.3461,
  "burst_score": 0.4287,
  "confidence": 0.3874,
  "attribution": "uncertain",
  "label": "Uncertain — May Be Human or AI"
}
```

The vocabulary signal correctly identifies casual short-word language
(vocab_score 0.35). The burstiness signal finds moderate variation (one
one-word sentence "underwhelming" mixed with longer ones). Combined confidence
of 0.39 lands in the "Uncertain" tier — the system says it doesn't know, which
is the right answer for a short ambiguous text.

The difference between these two cases — 0.74 vs 0.39 — shows the scoring
produces meaningful variation across the confidence range, not a constant.

**Tier 1 confirmation** (confidence = 0.26, "Likely Human-Written"):

To confirm the lowest tier is reachable, text with very short casual words and
varied sentence rhythm (fragmented thoughts, mix of 2-word and 18-word
sentences) produces:

```json
{
  "vocab_score": 0.1343,
  "burst_score": 0.3766,
  "confidence": 0.2554,
  "attribution": "likely_human",
  "label": "Likely Human-Written"
}
```

## Transparency Labels

The system returns one of four label strings based on the confidence score.
These strings are defined once in `detector.py:_label()` and are the same
string in every API response field, every audit log entry, and every status
lookup.

### Label 1 — Likely Human-Written (confidence < 0.30)

```
"Likely Human-Written"
```

What it means: both signals found characteristics consistent with human
authorship — casual or varied vocabulary and varied sentence rhythm. The system
leans toward human origin but the confidence value should always be read
alongside the label. A score of 0.28 is more human-like than a score of 0.05
would be, but both produce this label.

### Label 2 — Uncertain — May Be Human or AI (0.30 ≤ confidence < 0.50)

```
"Uncertain — May Be Human or AI"
```

What it means: the signals returned mixed evidence. The text may be
human-written, AI-generated, or AI-assisted in some sections. The system
cannot determine origin with meaningful confidence. Creators who receive this
label should not be treated as flagged — the system is saying it does not know.

### Label 3 — Likely AI-Assisted (0.50 ≤ confidence < 0.70)

```
"Likely AI-Assisted"
```

What it means: both signals lean toward AI patterns, but the score is not high
enough to call it definitive. The text shows formal vocabulary and/or uniform
sentence rhythm commonly seen in AI output. "AI-Assisted" is intentionally
hedged — the text might be fully AI-generated, heavily edited AI output, or
human writing that happens to be formal and uniform.

### Label 4 — Very Likely AI-Generated (confidence ≥ 0.70)

```
"Very Likely AI-Generated"
```

What it means: both signals scored strongly AI-like. The text shows low lexical
diversity, formal polysyllabic vocabulary, and/or highly uniform sentence
lengths simultaneously. This is the highest-confidence tier.

Four labels, not three: collapsing "Likely AI-Assisted" and "Very Likely
AI-Generated" into one tier would hide a meaningful difference. A creator at
0.72 confidence and a creator at 0.97 confidence should not receive identical
labels — the first is a judgment call, the second is a near-certainty.

## Appeals Workflow

When a creator believes their content was misclassified, they submit `POST
/appeal` with their `content_id` and a `creator_reasoning` explanation.

What happens on appeal:
1. The submission's status transitions from `"classified"` to `"under_review"`.
2. The global audit log entry for that content_id is updated in-place with
   `appeal_reasoning` and `appealed_at`. A reviewer checking `GET /log` sees
   the current state of the classification including the appeal.
3. A timestamped `appeal_submitted` event is appended to the per-submission
   event timeline (visible at `GET /audit/<content_id>`).
4. The original `confidence` and `label` are **not changed**. The reviewer
   sees the original classification alongside the creator's argument — overwriting
   the label before human review would defeat the audit trail.

## Rate Limits

The `/submit` endpoint is rate-limited to **10 requests per minute** and
**100 requests per day** per IP address. Exceeding either limit returns HTTP
429.

**Reasoning**: 10 per minute means one submission every 6 seconds. No human
produces original writing faster than that, so legitimate use is never blocked.
The per-minute cap prevents scripted flooding regardless of intent. The 100/day
cap allows thorough testing and real creative workflows (submitting multiple
chapters of a document) without enabling bulk classification of third-party
content.

### Rate limit evidence


## Known Limitations

### Formal human writing is systematically misclassified

An economics paper, legal brief, or academic abstract written by a human will
often receive "Likely AI-Assisted" or "Very Likely AI-Generated." This is a
direct consequence of how Signal 1 is built: the average word length and
long-word density sub-metrics reward any text that uses polysyllabic vocabulary
heavily — and formal human writing does exactly that. The signal cannot
distinguish "formal because legal writing requires precise multi-syllabic terms"
from "formal because an LLM statistically gravitates toward longer words."

This is not a calibration error. It is a fundamental property of the signal.
Fixing it would require a different kind of signal — one that reads semantic
coherence and factual grounding rather than surface vocabulary statistics.

### Short texts are classified as "Uncertain" regardless of actual origin

Any text under approximately 30 words returns `burst_score = 0.5` (fewer than
3 sentences, so CoV cannot be computed) and an unstable `vocab_score` (RTTR
is noisy at low N). Combined confidence lands near 0.5, which maps to "Likely
AI-Assisted." A tweet-length AI-generated text will escape classification; a
tweet-length human text will be called uncertain.

This is by design: the alternative — confidently misclassifying at low N — is
worse than saying uncertain. But it means the system has near-zero discriminative
power below ~30 words.

### Adversarial prompting defeats both signals

An LLM explicitly instructed to "write casually with short simple words and
varied sentence lengths" produces text that scores low on both signals. The
vocabulary richness and formality signals measure surface statistical properties,
not semantic coherence, factual grounding, or logical consistency. A
deliberately crafted LLM output can be tuned to defeat the statistical
fingerprints this system looks for. This is a known ceiling for heuristic
detection methods.

## Spec Reflection

### One way the spec guided implementation

Writing the uncertainty representation section of `planning.md` before any code
forced a concrete answer to a question I would otherwise have deferred: *what
does a confidence score of 0.6 actually mean?* Deciding in advance that 0.6 is
"Likely AI-Assisted — a hedge, not a conviction" shaped how I wrote the label
strings and explained why I set the tier boundaries where I did. During Milestone
4 testing, when all four inputs clustered between 0.40 and 0.47, I immediately
knew that was wrong — not because the numbers felt wrong, but because the spec
said "clearly AI-generated should score high" and these scores could not produce
that outcome. The spec gave me a precise diagnostic target, not just a vague
sense that something was off.

### One way implementation diverged from the spec

Signal 1 was specified in `planning.md` as "Vocabulary Richness (Root Type-Token
Ratio)" — a single metric. During Milestone 4 calibration, I discovered that
RTTR alone scored a formal AI paragraph and a casual human anecdote identically
(0.41 vs 0.40). The spec's own blind spot section had predicted this: "Short
texts (< 50 words) produce unreliable RTTR values." The prediction was right but
I had not internalized what it meant in practice. Testing showed that I needed to
extend Signal 1 to a composite of RTTR, average word length, and long-word
density before the scores separated meaningfully. The spec pointed to the
problem; calibration data confirmed it; the fix extended the signal architecture
beyond what was originally written.

## AI Usage

### Instance 1: Flask app skeleton and first signal function

I provided the detection signals section from `planning.md` (Signal 1
description with the RTTR formula and calibration reference points) and the
submission flow diagram to an AI tool and asked it to generate the Flask app
skeleton with `POST /submit`, `GET /health`, and the `_vocabulary_ai_score()`
function.

The AI produced a clean Flask skeleton with the RTTR formula implemented
correctly, in-memory dict storage keyed by UUID, and `jsonify` responses. The
signal function matched the spec formula exactly.

What I revised: I noticed the AI initialized `default_limits=["200 per day",
"50 per hour"]` on the Limiter — a generic catch-all that would also throttle
the `/log` and `/health` endpoints I hadn't built yet. I cleared `default_limits`
to `[]` and applied rate limits only to the specific routes that needed them
(`/submit` and `/appeal`), which is the correct approach for an API where
different endpoints have different abuse profiles.

### Instance 2: Appeal endpoint with incorrect field naming

I provided the appeals workflow section of `planning.md` (who can appeal, what
information they provide, the four-step system behavior, what a reviewer sees)
and the appeal flow diagram to an AI tool and asked it to generate the
`POST /appeal` endpoint.

The AI produced a working endpoint that updated status, mutated the audit log,
and returned the expected confirmation structure. However, it implemented the
endpoint as `POST /appeal/<content_id>` (path parameter) with a `reason` field
in the body — a reasonable interpretation of a REST resource URL, but not what
the milestone specification required.

What I overrode: I changed the endpoint to `POST /appeal` with `content_id` and
`creator_reasoning` in the request body, matching the milestone test curl command
exactly. I also changed how the appeal is reflected in `GET /log` — the AI had
appended a second log row for the appeal event, but the milestone spec said the
original entry should show `status: under_review` and `appeal_reasoning`. I
replaced the append with an in-place mutation of the existing global log entry,
searching by `content_id` inside the thread lock.
